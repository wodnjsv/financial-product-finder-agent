# 08. 의도 파싱 — N-best 해석·검증·되묻기·측정 루프

> 상태: 설계 초안 (2026-07-07) · 위치: [01-architecture.md](01-architecture.md) ① 프론트 LLM의 상세 · [06-query-planner.md](06-query-planner.md) `clarify_needed`의 격상판

## 0. 목표의 재정의

"완벽한 파싱"은 목표가 아니다 — 자연어는 본질적으로 모호하고, 사용자 본인도 자기 의도를 확정하지 못한 경우가 있다. 실전 목표는 세 가지의 조합이다:

1. **틀리기 어렵게**: 문제를 자유 생성이 아니라 닫힌 분류로 축소한다 (§1)
2. **틀리면 드러나게**: 파싱 결과를 그냥 믿지 않고 검증·에코백으로 노출한다 (§2, §4)
3. **드러나면 싸게 교정되게**: 모호성은 해석 메뉴로 사용자에게 넘기고, 교정은 데이터로 회수한다 (§3, §5)

의도의 ground truth는 사용자 머릿속에만 있다. 시스템의 일은 그걸 맞히는 게 아니라 **가장 싸게 꺼내는 것**이다.

## 1. 파서 구조: 닫힌 어휘 + N-best 출력

### 1.1 닫힌 어휘 (기존 원칙 유지)

파서(HyperClova X)는 슬롯 값을 자유 생성하지 않는다. [01 §4](01-architecture.md)의 슬롯 스키마에 정의된 값과 그래프 실재 노드 ID 중에서만 선택한다. 자유 텍스트 → 무한 오류 공간이, 닫힌 분류 → 유한 오류 공간이 된다.

### 1.2 표현 사전 (lexicon)

슬롯 값마다 실제 한국어 표현을 매핑한 사전을 유지하고 파서 프롬프트에 주입한다:

```yaml
# lexicon.yaml (발췌) — 실패 사례가 나올 때마다 여기에 추가 (모델 재학습 불필요)
risk_level.안정형: ["안전하게", "마음 편한", "묻어두고 싶은", "잃으면 안 되는", "안전빵"]
purpose.인컴:     ["따박따박", "월배당", "현금흐름", "이자 받으면서", "배당 받으면서"]
purpose.파킹:     ["잠깐 넣어둘", "비상금", "CMA처럼", "며칠만"]
purpose.성장:     ["올라탈", "수익 노리고", "공격적으로"]
```

이 사전이 파서의 도메인 지식이다. 프롬프트에는 사전 + few-shot 예시(골든셋의 어려운 케이스에서 발췌)가 들어간다.

### 1.3 N-best 후보 출력

파서는 단일 해석이 아니라 **해석 후보 목록**을 낸다:

```python
@dataclass
class ParseResult:
    candidates: list[Candidate]      # 신뢰도 내림차순, 최대 3개

@dataclass
class Candidate:
    slots: SlotSet                   # 01 §4 스키마 준수
    confidence: float                # 해석 전체 신뢰도
    slot_confidence: dict[str, float] # 슬롯별 신뢰도 (약한 슬롯 특정용)
```

출력은 JSON 스키마로 강제하고, 스키마 위반 시 자동 재파싱한다.

## 2. 3단 검증 (전부 코드, LLM 아님)

각 후보는 쿼리 플래너로 가기 전에 통과해야 한다:

| 단계 | 검사 | 실패 시 |
|---|---|---|
| **스키마 검증** | 형식·타입·필수 슬롯 존재 | 자동 재파싱 (1회) |
| **실재성 검증** | `views.target`이 그래프 실재 노드 ID인가 | 벡터 근사 매칭 fallback ([06 §4.3](06-query-planner.md)) |
| **논리 모순 검증** | 규칙으로 감지: 안정형+레버리지 요구, 파킹+장기, 최고등급+고수익 등 | 완화가 아니라 **되묻기**로 라우팅 — 모순은 파싱 오류이거나 의도 자체의 상충이므로 어느 쪽이든 물어야 한다 |

## 3. 라우팅: 언제 바로 가고, 언제 묻는가

핵심 규칙 — **"항상 묻기"가 아니라 "경합할 때만 묻기"**. 메뉴가 매번 뜨면 검색기가 아니라 설문지가 된다. 정상 상태에서 메뉴 발생률은 전체 질의의 10~20%를 목표로 한다.

```
                              ┌─ top1 신뢰도 ≥ 0.8 이고 2위와 격차 ≥ 0.3
                              │   → 바로 진행 + 에코백 (§4)          [대부분의 질의]
질의 → 파서 (N-best) → 검증 →─┤
                              ├─ 상위 후보 2~3개 경합 (격차 < 0.3)
                              │   → 해석 메뉴 제시 (§5)              [~10-20%]
                              │
                              └─ 전부 신뢰도 낮음 (top1 < 0.5) 또는 필수 슬롯 결측
                                  → 슬롯 단위 닫힌 질문               [소수]
```

임계값(0.8 / 0.3 / 0.5)은 v0 초기값 — 골든셋 축적 후 "메뉴 발생률 vs 오해석률" 트레이드오프로 튜닝한다.

- **멀티턴 상속**: 후속 질문("그중에 보수 낮은 걸로")은 직전 슬롯셋을 상속하고 바뀐 슬롯만 갱신한다.
- **사용자 프로파일 기본값**: 성향을 이미 아는 사용자는 `risk_level`을 묻지 않고 프로파일로 채우되, 에코백에 명시한다 ("평소 성향인 안정형 기준으로 찾았습니다").

## 4. 에코백 (echo-back) — 안 물은 질의의 안전망

바로 진행한 질의(80~90%)의 오류가 조용히 지나가지 않도록, **응답 첫머리에 시스템의 해석을 자연어로 되돌려준다**:

> "'안정적으로 배당 받으면서 반도체 회복에 올라탈' 상품 — **안정형 위험 수준에서, 분배금이 있고, 반도체 밸류체인에 노출된 ETF·펀드**로 찾았습니다."

- 응답 합성(HyperClova X)이 explanation trace의 슬롯·해석을 받아 생성한다. 완벽한 파서는 불가능해도, **틀린 파싱이 조용히 지나가는 일은 없앨 수 있다.**
- 사용자가 에코백을 보고 교정하면("아니, 공격적으로 가도 돼") 그 교정은 §6의 라벨 데이터로 회수된다.

## 5. 해석 메뉴 (disambiguation) — 모호성을 사용자에게 넘기기

경합 후보가 있을 때, 해석들을 사용자 언어로 번역해 고르게 한다. 사용자의 선택은 정의상 정답이고, 선택 = 재파싱이 필요 없는 오류율 0의 슬롯 확정이다.

### 5.1 옵션 생성 규약

1. **슬롯셋이 먼저, 문구는 번역**: 후보 슬롯셋(닫힌 어휘, 검증 통과분)을 먼저 확정하고, 각각을 사용자 언어로 번역한다. 옵션 문구 ↔ 슬롯셋은 1:1 — LLM이 옵션을 자유 작문하게 두면 옵션 자체가 환각일 수 있다. 선택 시 해당 슬롯셋이 **그대로** 플래너로 간다.
2. **차이가 나는 축만 보여준다**: 후보들이 실제로 갈리는 결정 축 하나를 잡아 2~3개 옵션으로. 슬롯 전체를 나열하지 않는다. 물을 가치의 기준 = **답에 따라 검색 결과가 크게 갈리는가** (듀레이션 한 스텝 차이는 물을 가치가 없다 — soft 필터가 처리).
3. **탈출구 필수**: "이 중에 없어요 (직접 설명할게요)" 옵션을 항상 포함. 자유 입력은 재파싱 경로로.
4. **라운드 제한**: 메뉴는 원칙적으로 질의당 1회. 두 번째 모호성은 top1으로 진행 + 에코백으로 처리.

### 5.2 예시

> **질의**: "안정적으로 배당 받으면서 반도체 회복에도 올라탈 수 있는 상품 있어?"
>
> 질문을 두 가지로 이해할 수 있어요. 어느 쪽에 가까우세요?
>
> **①** 배당도 주면서 반도체에도 노출된 **하나의 상품** (예: 반도체 비중이 있는 배당 ETF)
> **②** 안정적 배당 상품을 기본으로 하고, **반도체 상품을 따로 얹는** 조합
> **③** 이 중에 없어요 (직접 설명할게요)

①과 ②는 검색 결과가 완전히 달라지는 분기다 — ①은 단일 상품 검색, ②는 두 슬롯셋으로 분해된 복합 질의.

## 6. 측정·개선 루프 — 운영이 라벨링 공장이 된다

[05](05-analyst-structure.md)의 "측정 없는 강화는 없다"가 파서에도 그대로 적용된다.

### 6.1 골든셋

- **구축**: 한국어 질의 × 정답 슬롯셋 200~300쌍부터. 구성 비율 — 명확한 질의 40%, 은어·구어("따박따박", "몰빵은 싫어") 20%, 복합 의도 15%, 모순 10%, 어휘 밖 표현 15%.
- **용도**: 프롬프트·사전·few-shot을 바꿀 때마다 **회귀 테스트**. 감으로 프롬프트를 고치는 것은 도박이다.
- **집계는 슬롯별로 분해**: 어느 슬롯이 약한지 특정 (통상 `risk_level`·`views`가 어렵고 `product_types`는 쉽다).
- **오류 유형 구분 집계**: "모델이 틀림" vs **"스키마에 담을 자리가 없음"**을 구분한다. 후자가 많으면 고칠 것은 프롬프트가 아니라 슬롯 어휘다.

### 6.2 운영 데이터 자동 회수

| 이벤트 | 회수되는 것 |
|---|---|
| 해석 메뉴에서 선택 | `(질의, 선택된 슬롯셋)` — **정답 라벨이 공짜로 생성됨** |
| 에코백 후 사용자 교정 | `(질의, 교정된 슬롯셋)` 라벨 |
| "이 중에 없음" 선택 | **스키마 갭 신호** — 비율을 대시보드로 추적, 높으면 슬롯 어휘 확장 검토 |
| 검색 결과 0개 (완화 소진) | 검토 큐 — 파싱 오류인지 유니버스 한계인지 분류 |

이 루프가 돌면: 선택 데이터 축적 → 파서 개선(프롬프트→파인튜닝) → 모호성 감소 → 메뉴 발생률 감소의 선순환.

### 6.3 파인튜닝 전환 기준

프롬프트+사전 개선이 골든셋 정확도에서 한계에 닿고, 라벨 데이터가 수천 건 쌓였을 때만 검토한다. 그 전에는 사전·few-shot 갱신이 더 싸고 빠르다.

## 7. 전체 흐름 요약

```
질의 → 파서 (닫힌 어휘 + 사전 + N-best) → 3단 검증
     → 라우팅: ┌ 확신 → 진행 + 에코백 ──────────────┐
               ├ 경합 → 해석 메뉴 (선택=슬롯 확정) ──┤→ 쿼리 플래너 (06)
               └ 불명 → 슬롯 단위 닫힌 질문 ─────────┘
     [모든 선택·교정 → 골든셋 회수 → 파서 개선 (§6)]
```

## 8. Phase별 도입

| Phase | 범위 |
|---|---|
| 1 | 단일 파스 + 스키마·실재성 검증 + 에코백 + **골든셋 200개 구축과 회귀 테스트** (메뉴 없이 시작) |
| 2 | N-best 출력 + 신뢰도 라우팅 + 해석 메뉴 + 논리 모순 검증 + 운영 라벨 자동 회수 |
| 3 | 임계값 튜닝(메뉴 발생률 vs 오해석률), 라벨 수천 건 축적 시 파인튜닝 검토 |
