# 05. 애널리스트 내부 구조와 학습 루프 — 두 축 설계

> 상태: 설계 초안 (2026-07-06) · 적용 대상: 12명 애널리스트 전원 ([02-sector-analysts.md](02-sector-analysts.md))

## 1. 설계 원리: 실력은 두 축이다

애널리스트의 실력을 두 축으로 분리하고, 각 축에 서로 다른 학습 루프를 붙인다.

| 축 | 내용 | 피드백 속도 | 측정 | 학습 메커니즘 |
|---|---|---|---|---|
| **크래프트 (절차)** | 데이터를 어디서 어떻게 가져오고, 모델을 어떻게 갱신하고, 분석을 어떻게 구조화하는가 | 즉각 (실행 시점) | 절차 성공: 데이터를 제대로 뽑았나, grounding 검증을 통과했나 | 헤르메스 패턴 그대로 — 경험 기반 스킬 생성·자기개선 |
| **판단 (예측)** | 그 분석의 결론이 맞았는가 — 뷰가 예측적으로 옳았는가 | 지연 (주~월) | 시장 결과: 뷰 적중률·IC | **우리가 짓는 평가 파이프라인** → 헤르메스 큐레이션에 보상으로 주입 |

두 축은 대체재가 아니라 보완재다. 헤르메스 루프만 돌리면 에이전트는 "정확한" 쪽이 아니라 "과업을 매끄럽게 해내는 / 자신 있게 들리는" 쪽으로 드리프트한다. **판단 축의 정확도 신호가 크래프트 축의 큐레이션을 제약해야** 두 루프가 하나의 강화로 합쳐진다.

## 2. 애널리스트 내부 구조 (공통)

```
┌─ 섹터 애널리스트 에이전트 ──────────────────────────────┐
│                                                      │
│  정체성 프롬프트   ← 02문서의 담당 범위·밸류체인·지표에서 생성  │
│                                                      │
│  스킬 라이브러리 (크래프트 축)                            │
│   skill_id · version · 절차 성공률 · 판단 기여 점수        │
│   예: "DRAM 고정가 갱신 → 메모리 세그먼트 뷰 재평가"        │
│                                                      │
│  리서치 메모리 (자기 큐레이션)                            │
│   note_id · 신뢰 점수 · 예: "과거 재고 사이클에서 X 패턴"    │
│                                                      │
│  출력: AnalystView (03 스키마)                          │
│   + provenance: 이 뷰에 쓰인 skill_ids, note_ids  ★귀속용 │
└──────────────────────────────────────────────────────┘
```

핵심 규약은 마지막 줄이다: **모든 뷰는 자신을 만든 스킬·노트의 ID를 기록한다.** 이 provenance가 없으면 지연된 성과를 스킬·메모리로 역전파할 방법이 없다.

- **스킬** = 재사용 가능한 분석 절차 (프롬프트 + 도구 호출 시퀀스). 예: "이 반도체 기업 실적에서 메모리 ASP 추세 추출", "임상 등록부 확인 후 승인확률 재계산". 반복 수행에서 자동 생성되고 사용 중 개선된다 (헤르메스 패턴).
- **메모리** = 누적 리서치 노트. "지난 분기에 이 회사 재고 적체를 플래그했고 그 논리가 맞았다/틀렸다" — 주기적 큐레이션(요약·강등·보관)을 에이전트가 직접 수행한다.
- **사용자 모델링은 여기 없다.** 헤르메스의 사용자 모델링 조각은 섹터 애널리스트가 아니라 **프론트 HyperClova X 단의 사용자 프로파일**(리스크 성향·선호 파악)로 이동한다. 애널리스트는 시장만 본다.

## 3. 빠른 루프 — 크래프트 (즉각 피드백)

트리거: 스킬 실행 시점마다.

| 신호 | 판정 |
|---|---|
| 데이터 추출 정합성 | 뽑은 수치가 원본과 일치하는가 (자동 대조 가능한 것 우선) |
| grounding 검증 통과율 | 생성한 그래프 엣지 제안·뷰 근거가 검증 게이트(03 §5)를 통과했는가 |
| 실행 오류율 | 도구 호출 실패, 소스 접근 불가 등 |

실패 시: 헤르메스 방식의 스킬 자기개선(실패 원인 분석 → 스킬 개정 → 버전 증가). 이 루프는 절차의 품질만 다루며, **뷰의 방향이 맞았는지는 판정하지 않는다.**

## 4. 느린 루프 — 판단 (지연 피드백)

### 4.1 보상 신호 정의

| 신호 | 정의 | 역할 |
|---|---|---|
| **뷰 적중 (주 신호)** | 뷰 시점 T의 스탠스 방향 × 담당 세그먼트 노출 바스켓의 T+20/T+60 영업일 **상대수익률**(시장 대비) 부호 | 판단 축의 유일한 1차 신호 |
| **IC (정보계수)** | 같은 애널리스트의 뷰 강도(confidence×방향)와 실현 상대수익률의 순위 상관 (뷰가 쌓인 뒤) | 적중률보다 강건한 장기 지표 |
| **랭킹 채택률 (보조)** | 이 애널리스트의 뷰가 가중된 추천을 사용자가 채택한 비율 | **보조로만.** 인기 편향이 있어 판단 신호로 삼으면 "그럴듯한 쪽"으로 드리프트한다 |

세그먼트 노출 바스켓 = 그래프에서 `OPERATES_IN` 해당 세그먼트인 기업들의 시가총액 가중 바스켓. **뷰 시점의 그래프 상태로 구성한다** — 03문서의 시간성 원칙(valid_from/to)이 이 계산의 전제다.

상대수익률의 벤치마크: v0은 KOSPI(코스닥 기업 비중 높은 세그먼트는 혼합지수). 시장 전체의 등락을 제거해 "섹터 판단"만 남기기 위함이다.

### 4.2 지연 귀속 (attribution)

```
뷰 기록 (T):     AnalystView { stance, confidence, as_of=T,
                              skill_ids=[s1,s2], note_ids=[n3] }
평가 (T+20/60):  바스켓 상대수익률 r 계산 → 뷰 점수 v = sign(stance)·sign(r)·|r| 기반
역전파:          v를 s1, s2, n3에 분배 (v0: 균등 분배)
```

- 평가 시점은 T+20(1개월)과 T+60(3개월) 두 개를 병행 기록 — 뷰의 시계(horizon)별 적중이 다를 수 있다.
- 귀속 분배는 v0에서 균등. 정교화(뷰 생성 시 에이전트가 기여도를 자가 신고)는 데이터가 쌓인 뒤 검토.
- 뷰가 없으면 귀속도 없다: **뷰를 내지 않은 기간은 페널티도 보상도 없음** → 단, 뷰 생산 빈도 하한을 스케줄(크론)로 강제해 "안 내고 버티기"를 차단.

### 4.3 신호 → 가중치 변환

| 대상 | 갱신 규칙 | 결과 |
|---|---|---|
| **스킬** | 판단 기여 점수 = 그 스킬이 기여한 뷰들의 적중 EMA(최근 가중) | 임계 미달 → 개정 큐: "이 스킬이 낳은 뷰들이 반복적으로 틀렸다. 절차의 어떤 가정이 문제인가"를 자기개선 프롬프트로 주입 |
| **메모리 노트** | 신뢰 점수를 같은 방식으로 갱신 | 저신뢰 노트는 큐레이션 시 강등·보관 (삭제하지 않음 — 반례 기록도 자산) |
| **애널리스트 자체** | 애널리스트별 적중률·IC → **랭킹 엔진의 뷰 가중 계수** | 적중률 낮은 애널리스트의 뷰는 랭킹에 덜 반영 (시스템 차원 안전장치 — 개별 스킬 개선과 독립적으로 작동) |

### 4.4 과최적화·드리프트 방지 규칙

1. **표본 하한**: 평가된 뷰가 N개(v0: 10) 미만이면 가중치 갱신 보류 — 소표본 잡음으로 스킬을 죽이지 않는다.
2. **grounding 강제 유지**: 뷰의 rationale은 Source 참조 필수 (빠른 루프가 검사). 정확도 보상이 있어도 근거 없는 뷰는 애초에 기록되지 않는다.
3. **자기개선 트리거 제한**: 스킬 개정은 절차 실패 또는 판단 실패 신호가 있을 때만. "더 매끄럽게" "더 자신 있게"는 트리거가 아니다.
4. **confidence 캘리브레이션 점검**: confidence 구간별 실제 적중률을 분기마다 대조 — 과신 드리프트를 조기 감지.

## 5. 두 루프의 결합 (전체 그림)

```
        ┌────────────── 빠른 루프 (즉각) ──────────────┐
        │                                            │
 스킬 실행 → 절차 성공/실패 → 스킬 자기개선 (헤르메스 패턴)     │
        │                                            │
        ▼                                            │
 뷰 생산 (provenance 기록) ──→ 랭킹 가중 ──→ 추천        │
        │                                            │
        │         ┌────── 느린 루프 (지연, 우리가 지음) ──┴──┐
        ▼         ▼                                       │
 T+20/60 성과 평가 → 뷰 적중·IC → 스킬·노트 가중치 역전파 ──────┘
                        │
                        └→ 애널리스트별 랭킹 가중 계수 갱신
```

헤르메스가 큐레이션 기계(스킬 자기개선·메모리 넛지·스케줄 실행)를 제공하고, 느린 루프가 그 기계에 올바른 보상(예측 정확도)을 먹인다. 이 연결이 없으면 크래프트만 늘고 판단은 늘지 않는다.

## 6. 단계별 도입 (01문서 Phase와의 정합)

| Phase | 이 문서에서 도입되는 것 |
|---|---|
| 1 | 없음 (애널리스트 미가동). 단, **뷰·랭킹 로그 스키마는 이때부터** — provenance 필드 포함해 설계 |
| 2 | 애널리스트 가동 + 빠른 루프(크래프트) + 뷰 provenance 기록. 느린 루프는 **기록만 하고 갱신은 안 함** (표본 축적) |
| 3 | 느린 루프 가동: 적중 평가 → 가중치 역전파 → 랭킹 가중 계수. 캘리브레이션 대시보드 |
