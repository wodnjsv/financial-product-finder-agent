# 01. 전체 아키텍처 — 금융 상품 탐색 에이전트

> 상태: 설계 초안 (2026-07-06) · 범위: 국내 상장 ETF·펀드·채권

## 1. 시스템 목표

사용자의 한국어 자연어 질의(예: "안정적으로 배당 받으면서 반도체 업황 회복에 올라탈 수 있는 상품")를 받아,
**실존하는 국내 상장 금융상품 DB에서 조건에 맞는 상품을 검색·랭킹하고, 근거와 함께 한국어로 추천**한다.

포트폴리오 비중 최적화기가 아니다. 검색·추천(retrieval + ranking) 시스템이다.

## 2. 불변 원칙

1. **LLM은 상품을 직접 고르지 않는다.** 의도 구조화(입구)와 결과 설명(출구)만 담당한다.
   상품 매칭·랭킹은 실제 데이터에 대한 결정적(deterministic) 쿼리로만 수행한다. → 환각 상품 추천 원천 차단.
2. **근거 없는 엣지는 그래프에 커밋하지 않는다.** 에이전트가 생성하는 모든 관계는 출처·신뢰도를 달고 검증을 통과해야 한다.
3. **모든 데이터는 point-in-time.** 엣지는 `valid_from`/`valid_to`를 갖고, 뷰·추천은 시점별로 기록해 사후 성과 측정이 가능해야 한다.
4. **측정 없는 강화는 없다.** 강화 루프는 평가 파이프라인(적중률·채택률)이 먼저 존재해야 작동한다.

## 3. 컴포넌트 구성

```
사용자 질의 (한국어)
   │
   ▼
┌─────────────────────────┐
│ ① 프론트 LLM (HyperClova X) │  의도 파싱 → 슬롯
└─────────────────────────┘
   │  {목적, 리스크, 기간, 섹터뷰, 상품군, 제약}
   ▼
┌─────────────────────────┐
│ ② 쿼리 플래너              │  슬롯 → 그래프 질의 + DB 필터 (결정적 변환)
└─────────────────────────┘
   │
   ▼
┌─────────────────────────┐     ┌──────────────────────────┐
│ ③ 검색·랭킹 엔진           │ ◄── │ ⑤ 섹터 애널리스트 레이어     │
│  하드필터→노출도→뷰 가중    │     │  11 섹터 + 1 매크로·금리    │
└─────────────────────────┘     └──────────────────────────┘
   │                                  ▲          │
   ▼                                  │          ▼
┌─────────────────────────┐     ┌──────────────────────────┐
│ ① 프론트 LLM (응답 합성)    │     │ ⑥ 강화 루프 (헤르메스 패턴)  │
└─────────────────────────┘     │  뷰 기록→성과 대조→스킬 갱신 │
   │                            └──────────────────────────┘
   ▼
추천 결과 + 근거 설명 (한국어)

  ── 저장소 (④ 폴리글랏) ──────────────────────────────
  지식 그래프 │ 시계열 │ 벡터(RAG) │ 관계형(상품 마스터)
  ── 데이터 수집 (⑦) ─────────────────────────────────
  ECOS · FRED · KIS · 토스증권 · DART · KRX · 금투협 …
```

### ① 프론트 LLM (HyperClova X)

- **의도 파싱**: 질의 → 슬롯 구조체(아래 §4). 함수 호출/JSON 강제 출력.
- **응답 합성**: 랭킹 결과 + 근거 데이터 → 자연스러운 한국어 설명. 결과를 바꾸지 않고 설명만 한다.
- 모호한 질의는 되묻기(슬롯 미충족 시 명확화 질문 생성).

### ② 쿼리 플래너

슬롯을 두 종류의 질의로 변환한다:
- **하드 필터** (관계형 DB): 상품군, 신용등급, 듀레이션 범위, 총보수 상한, 분배금 유무, 최소 AUM 등.
- **그래프 질의** (지식 그래프): 섹터뷰/테마 → 밸류체인 세그먼트 → 노출 상품 경로 탐색.

LLM이 아닌 규칙 기반 변환기다. 슬롯 값 → 쿼리 매핑 테이블이 이 컴포넌트의 실체이며, 제품 품질의 절반이 여기에 있다.

### ③ 검색·랭킹 엔진

3단 점수 조합:
1. **하드 필터**: 통과/탈락 (점수 아님).
2. **그래프 노출도 점수**: 질의된 섹터뷰·테마와 상품의 밸류체인 노출이 얼마나 정합하는지.
   상품→구성종목→기업→세그먼트 경로의 가중 합 (구성 비중 × 세그먼트 매칭 강도).
3. **애널리스트 뷰 가중**: 해당 세그먼트/섹터에 대한 담당 애널리스트의 현재 스탠스(강세/중립/약세, 신뢰도)로 점수 보정.

최종 점수와 함께 **점수 분해 내역(explanation trace)**을 반환한다 — 응답 합성과 사후 평가 모두 이걸 소비한다.

### ④ 폴리글랏 저장소

| 저장소 | 역할 | 후보 기술 |
|---|---|---|
| 지식 그래프 | 상품→종목→기업→섹터→밸류체인→원자재 관계 | Neo4j (프로퍼티 그래프, Cypher) |
| 시계열 | 가격·NAV·금리·거래량 이력 | TimescaleDB 또는 PostgreSQL 파티션 |
| 벡터 | 공시·리포트·뉴스 원문 RAG | pgvector (초기), 규모 커지면 전용 벡터DB |
| 관계형 | 상품 마스터·속성 (하드 필터용) | PostgreSQL |

초기엔 **PostgreSQL 하나(+pgvector, 파티션) + Neo4j 하나**의 2-DB 구성으로 충분하다. 그래프 노드는 다른 저장소의 데이터를 ID로 참조만 한다.

### ⑤ 섹터 애널리스트 레이어

GICS 11개 섹터 애널리스트 + **매크로·금리 애널리스트 1명(12번째)**.

> 12번째를 추가하는 이유: 채권 상품 추천에는 금리 커브·크레딧 스프레드 분석이 필수인데 GICS 섹터로는 커버되지 않는다. 또한 금리·환율·유가는 전 섹터에 전파되는 공유 컨텍스트라서 각 섹터 에이전트가 중복 분석하는 것보다 한 곳에서 생산해 배포하는 게 맞다.

각 애널리스트의 산출물(전부 시점 기록):
- **뷰(View)**: 담당 세그먼트별 스탠스 {강세/중립/약세} + 신뢰도 + 근거 요약 + 유효기간.
- **그래프 갱신 제안**: 새 관계 발견 시 출처를 달아 제안 → 검증 게이트 통과분만 커밋.
- **모니터링 노트**: 담당 지표의 이상 신호 (스케줄 실행).

상세 설계: [02-sector-analysts.md](02-sector-analysts.md)

### ⑥ 강화 루프 (헤르메스 패턴 차용)

[nousresearch/hermes-agent](https://github.com/nousresearch/hermes-agent)에서 차용하는 것은 코드가 아니라 세 가지 메커니즘이다:

1. **경험 기반 스킬 생성·개선**: 애널리스트가 반복 수행하는 분석 절차(예: "정제마진 스프레드 계산 후 정유주 뷰 갱신")를 재사용 가능한 스킬(프롬프트+도구 호출 시퀀스)로 축적하고, 성과 신호에 따라 개정한다.
2. **자기 큐레이션 메모리**: 섹터별 지식(과거 사이클 패턴, 검증된 인과관계)을 에이전트가 직접 정리·갱신하는 메모리로 유지한다.
3. **스케줄 기반 자율 실행**: 크론 스케줄로 정기 뷰 갱신·지표 모니터링을 돌린다 (사용자 질의와 무관하게).

폐순환: **뷰/추천 기록 → 실제 성과·사용자 채택률 대조 → 적중률 낮은 스킬·메모리 개정 → 더 나은 뷰**.
전제조건인 평가 파이프라인: 모든 뷰·추천·랭킹 점수를 시점과 함께 로그 → T+20일/60일 상품 성과와 대조.

## 4. 슬롯 스키마 v0

```yaml
intent:
  purpose:        # 인컴(배당·이자) | 성장 | 절세 | 파킹(단기자금) | 헤지
  risk_level:     # 안정형 | 중립형 | 공격형  (제약으로 변환: 등급·변동성·듀레이션)
  horizon:        # 단기(<1y) | 중기(1-3y) | 장기(>3y)
  views:          # [{target: 섹터|세그먼트|테마|원자재, direction: 강세|약세, strength}]
  product_types:  # [ETF, 펀드, 채권] (미지정 시 전체)
  constraints:    # {max_fee, min_dividend_yield, max_duration, min_credit_rating,
                  #  exclude: [세그먼트|기업], currency_hedge, min_aum}
  clarify_needed: # 슬롯 부족 시 되물을 질문 목록
```

슬롯 → 쿼리 매핑 테이블(예: "안정형" → 채권은 AA- 이상·듀레이션 5년 이하, ETF는 변동성 하위 50% 등)은
구현 단계에서 별도 문서로 확정한다. **이 매핑이 제품 품질을 결정하는 1순위 작업이다.**

## 5. 구현 단계 (MVP 절단선)

| Phase | 범위 | 성공 기준 |
|---|---|---|
| **1** | 상품 마스터 수집 + 그래프 정적 계층 구축 + 의도 파싱 + 하드필터 + 그래프 노출도 랭킹 | "반도체 소재 노출 ETF 찾아줘" 류 질의에 실존 상품만, 근거 경로와 함께 반환 |
| **2** | 섹터 애널리스트 뷰 생산 + 뷰 가중 랭킹 + 뷰 시점 기록 | 뷰 반영 전/후 랭킹 차이를 설명 가능; 모든 뷰가 로그에 남음 |
| **3** | 평가 파이프라인 + 강화 루프 (스킬·메모리 갱신) | 애널리스트별 적중률 대시보드; 저성과 스킬 자동 개정 이력 |

Phase 1은 애널리스트 에이전트 없이도 유용해야 한다 — 콜드스타트 시점에 강화 루프는 존재하지 않는다.
